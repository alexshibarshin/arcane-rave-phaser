## Архитектурные принципы

### EventBus — единый канал коммуникации

Системы и сцены обмениваются сигналами через типизированный `EventBus` (`src/events/EventBus.ts`).
Все события объявлены в `EventMap`. Используй обёртки `emit`, `on`, `off`, `once`.

### Конфиги в `src/config/`

Все runtime-значения (размеры, баланс, ключи сцен, параметры волн, визуальные настройки) лежат в `src/config/`.
Не размазывай магические числа по коду.

### GameScene: фиксированный порядок фаз кадра

```text
1. input systems
2. simulation systems
3. root entities
4. object pools
5. cleanup hooks
```

Новая система должна явно относиться к одной из фаз и не зависеть от результатов более поздних фаз.

### ObjectPool — инфраструктура, не логика

Пул (`src/utils/ObjectPool.ts`) универсален. Не знает ничего про тип игры, только контракт `IRenderable`.

## Правила для тестов

Тесты проверяют поведение, инварианты и контракты, а не дублируют текущие значения конфигов.

**Нельзя:**

- assert'ить конкретные числа из `src/config/`, если смысл теста не в самом контракте конфигурации
- фиксировать тайминги, радиусы, урон, цены, оффсеты, counts, цвета, id волн — только потому, что «сейчас так»
- превращать тест в snapshot баланса или authored-контента

**Нужно:**

- проверять инварианты: ссылки валидны, значения положительные, диапазоны корректны, состояния переходят правильно
- проверять поведение при изменении конфигов, а не совпадение с текущими числами
- использовать сравнения вида «увеличилось/уменьшилось/не пусто/применился эффект», если точное число не часть публичного контракта

Если при тюнинге конфига тест падает без поломки логики — тест написан неправильно, его нужно ослабить или удалить.
