# Combat Sandbox — Design Spec

## Feature Overview

`Combat Sandbox` — это первая playable-реализация боевой фазы `Arcane Rave`. Фича нужна не для проверки всего `stage`-цикла, а для максимально ранней валидации боевого ядра:

- непрерывное вращение `record`;
- активации слотов под фиксированной `needle`;
- перенос и потребление `note packet`;
- базовое давление врагов на `base`;
- читаемость мобильного боевого экрана в `9:16`.

Эта фича должна ответить на четыре вопроса:

1. Читается ли расположение объектов на экране во время боя.
2. Работает ли core fantasy “боевой винил вращается под иглой”.
3. Понятно ли игроку, как порядок слотов влияет на `note packet` и итоговый урон.
4. Достаточно ли выразителен визуальный язык первого primitive-based прототипа.

Это не временный “prototype scene” с мусорным неймингом. Это первая прод-названная боевая сцена и первый боевой runtime, которые потом будут расширяться в сторону полного `combat phase`.

## Core Mechanics

### Combat Loop

1. Игрок открывает бой.
2. Бой стартует в состоянии `preview`.
3. На экране уже собраны все ключевые боевые элементы:
   - `record`;
   - `needle`;
   - `base / DJ booth / capybara`;
   - preset пешек на пластинке;
   - верхний HUD.
4. Идет короткая пауза на чтение экрана.
5. Бой переходит в состояние `running`.
6. `record` начинает непрерывно вращаться против часовой стрелки.
7. Когда центр сектора пересекает `needle`, слот активируется.
8. Активированная `pawn` взаимодействует с текущим `note packet`, выбирает цель и наносит урон.
9. В это же время сверху вниз спавнятся враги, движутся к базе и создают давление.
10. Бой заканчивается в `victory`, когда убиты все враги, которые должны были заспавниться.
11. Бой заканчивается в `defeat`, когда `base HP <= 0`.
12. После `victory` или `defeat` доступен `Restart`, который полностью перезапускает боевые сцены.

### Record, Needle, Rotation

- У `record` ровно `8` слотов.
- `slot 0` находится в позиции `12:00` под `needle`.
- Остальные слоты нумеруются по часовой стрелке.
- Пластинка вращается против часовой стрелки.
- Это значит:
  - текущий активный слот виден сверху;
  - уже прошедшие слоты уходят налево;
  - следующие слоты подходят справа.
- `needle` закреплена строго на `12:00`.
- `needle` не находится вне пластинки. Она визуально тянется от верхней части базы вверх к краю `record`.
- Логика активации завязана не на “игла вошла в сектор”, а на пересечение `needle` с `center line` сектора.
- Скорость вращения задается конфигом.
- Стартовый baseline: `1` активация в секунду, то есть `8` секунд на полный оборот.
- Стартовый угол пластинки задается явно в конфиге `wave`. По умолчанию `slot 0` центром стоит под `needle`.

### Slot Structure

Каждый слот — это не точка, а полноценный сектор с двумя зонами:

- `outer pawn zone`
  - зона ближе к внешнему краю пластинки;
  - здесь физически стоит `pawn`;
  - здесь же находится pedestal и `tier`-звезды;
- `inner rule zone`
  - зона ближе к центру пластинки;
  - здесь отображается правило работы с `note` для текущей `pawn`.

Правила структуры слота:

- Слот имеет явную заметную рамку.
- `empty slot` тоже имеет рамку и обе зоны.
- Деление по радиусу стартово принимается `50 / 50`.
- Из-за клиновидной формы сектора внутренняя зона все равно имеет меньшую площадь, чем внешняя.
- Внутренняя зона визуально должна ощущаться как напечатанный лейбл на виниле.

### Slot Activation Feedback

В момент активации слота должен происходить трехслойный trigger feedback:

1. Ярко вспыхивает рамка сектора.
2. `pawn construct` внутри сектора дает короткий pulse/glow.
3. `inner rule zone` тоже кратко подсвечивается.

Для `empty slot`:

- срабатывает только вспышка сектора и его пустой площадки;
- атаки и движения нот нет.

### Note Packet Rules

- В системе существует один текущий `note packet`.
- Пакет всегда содержит `note` только одного цвета.
- Вместимость пакета ограничена `5` нотами.
- Если нот стало бы больше `5`, лишние сгорают.
- Пакет сбрасывается в конце `wave`.
- Визуально пакет показывается прямо над головой капибары.
- Отображение пакета — это `0..5` реальных цветных значков `note`, а не абстрактный счетчик.

### Note Visual Language

- В игре должен быть один канонический визуальный объект `note`.
- Это классический glyph восьмой ноты с хвостиком.
- Этот glyph переиспользуется везде:
  - в центре над капибарой как `note packet`;
  - в полете между `pawn` и центром;
  - внутри `inner rule zone` как часть правила;
  - потенциально в будущих UI-подсказках.
- Каждая `note` окрашивается в нужный цвет системы стихий.
- Ноты, висящие над капибарой, имеют медленную плавную bounce-анимацию по `Y`.

### Generator Rules

- `generator` всегда наносит свой базовый урон.
- Если входящий `note packet` совпадает по цвету:
  - `generator` добавляет `2` ноты своего цвета;
  - соблюдается лимит вместимости;
  - цвет пакета не меняется.
- Если входящий `note packet` другого цвета:
  - старый пакет ломается;
  - визуально старые ноты гасятся/рассыпаются;
  - создается новый пакет из `2` нот цвета `generator`.
- Внутри `inner rule zone` правило `generator` отображается как `+♪♪`.
- Все ноты внутри этой надписи окрашены в цвет самой `pawn`.

### Finisher Rules

- `finisher` всегда поглощает ноты собственного цвета.
- Отдельный параметр `consumedColor` не нужен.
- Если входящий `note packet` совпадает по цвету:
  - `finisher` поглощает все ноты пакета;
  - итоговый урон модифицируется глобальной таблицей множителей по количеству поглощенных нот.
- Если входящий `note packet` другого цвета:
  - пакет ломается;
  - `finisher` все равно атакует;
  - итоговый множитель считается по `0` поглощенным нотам.
- После атаки `finisher` выпускает `1` новую выходную `note`.
- `outputNoteColor` задается явно в данных конкретной пешки.
- `outputNoteColor` всегда должен отличаться от собственного цвета `finisher`.
- Внутри `inner rule zone` правило `finisher` отображается как `-all ♪ -> +♪`.
- Все ноты внутри надписи окрашены в соответствующие цвета.

### Finisher Damage Multipliers

Финишеры используют одну глобальную таблицу множителей по количеству поглощенных нот:

- `0 => 0.75`
- `1 => 1.0`
- `2 => 1.15`
- `3 => 1.4`
- `4 => 1.75`
- `5 => 2.25`

Итоговая формула:

- `generatorDamage = baseDamage`
- `finisherDamage = baseDamage * finisherConsumedNotesMultiplier[consumedNotes]`
- после этого, если сработала слабость по цвету, применяется `x1.5`

### Color System and Weakness

- В первой версии сразу работают `red`, `green`, `blue`.
- Ось цвета едина для:
  - `pawn`;
  - `note`;
  - атак;
  - врагов.
- Базовое правило слабостей:
  - `red > green`
  - `green > blue`
  - `blue > red`
- При выгодном матчапе урон получает множитель `1.5`.
- При нейтральном и невыгодном цвете штрафа нет.
- Резистов нет.

### Pawn Targeting

- Все `pawn` в первой версии используют одно и то же правило таргетинга.
- При активации `pawn` выбирает ближайшего врага к себе по обычной мировой дистанции.
- Считается Euclidean distance между мировой позицией `pawn` и мировой позицией врага.
- Ограничения дальности нет.
- Если врагов нет, атака уходит в пустоту и система не ломается.

### Attack Presentation

Атаки `pawn` в первой версии:

- не используют проджектайлы;
- происходят как мгновенный hit effect;
- визуально представлены неоновым beam / pulse;
- имеют выраженный `glow`;
- не обязаны быть идеально прямыми;
- могут быть ломанными или иметь молниеподобные всполохи.

При усиленном `finisher`:

- beam становится ярче;
- beam становится толще;
- beam получает более сильный glow;
- hit feedback на враге становится более заметным.

### Enemy Rules

- В первой версии есть один базовый архетип врага.
- У него есть три цветовых варианта: `red`, `green`, `blue`.
- Это один behavioral archetype, а не три разных паттерна поведения.
- После спавна враг:
  - получает случайный `spawnX`;
  - начинает двигаться строго по `Y` вниз;
  - не меняет `X` по пути.
- Враги могут накладываться друг на друга.
- Сепарации, расталкивания и avoidance-системы нет.
- Когда дистанция от врага до `base center` становится меньше либо равна `attackRange`, враг останавливается.
- После короткой логической остановки он начинает атаковать базу с собственным `attackCooldown`.
- Каждый удар наносит `attackDamage`.
- В будущем могут появиться ranged-враги и проджектайлы, но эта итерация их не включает.

### Wave and Sub-wave Rules

- Первая реализация содержит одну `wave`.
- Верхний HUD поэтому показывает `Wave 1/1`.
- Одна `wave` состоит из одной или нескольких `sub-wave`.
- У каждой `sub-wave` есть:
  - абсолютный `startTimeMs` от старта `wave`;
  - `spawnIntervalMs`;
  - состав в формате `enemyId -> count`.
- При активации `sub-wave` разворачивает состав в `spawn bag`.
- `spawn bag` — конечный мешок врагов для этой `sub-wave`.
- На каждом тике спавна из мешка случайно достается один враг без возврата.
- Если две `sub-wave` активны одновременно, они спавнят независимо друг от друга.
- Победа не может произойти, пока:
  - не активировались все `sub-wave`, которые должны стартовать по расписанию;
  - не опустели все `spawn bag`;
  - не умерли все живые враги на поле.

### Spawn Positioning

- Враги могут спавниться за пределами видимой верхней границы экрана.
- Зона спавна по `X` почти на всю ширину экрана.
- `spawnX` выбирается случайно из диапазона.
- Чтобы уменьшить визуальное слипание, используется простая защита от слишком близких соседних `spawnX`.
- При неудачном наборе попыток можно брать fallback-значение, чтобы спавн не зависал.

### Pause and Restart

- `Pause` в первой версии функциональный, а не декоративный.
- При паузе останавливаются:
  - вращение `record`;
  - движение врагов;
  - атаки врагов;
  - расписание `wave` и `sub-wave`;
  - боевые таймеры;
  - боевые VFX с логическим таймингом.
- На экране показывается overlay:
  - `Paused`
  - `Resume`
  - `Restart`
- `Restart` всегда делает полный restart боевых сцен.
- Soft reset боевого runtime вручную не используется.

## Implementation Decisions

### Systems Touched

- боевая сцена;
- HUD-сцена;
- runtime-модель боя;
- state machine боя;
- рендер `record`;
- рендер `pawn`;
- рендер и симуляция врагов;
- `wave / sub-wave / spawn bag`;
- `note packet`;
- боевые VFX;
- верхний HUD и overlay;
- `EventBus`;
- конфиги layout, balance и content.

### Architecture

#### Scene Ownership

- `CombatScene`
  - владеет всем боевым runtime;
  - отвечает за создание боевого мира;
  - обновляет симуляцию;
  - отвечает за layout world-элементов;
  - публикует события для HUD.

- `HUDScene`
  - отвечает только за UI и overlay;
  - не владеет боевой симуляцией;
  - не пересчитывает боевые состояния;
  - показывает полученные через события данные:
    - `Pause`;
    - `Wave 1/1`;
    - `Enemies left`;
    - `Paused`;
    - `Victory`;
    - `Defeat`.

- Scaffold `UIScene`
  - остается scaffold-элементом проекта;
  - не должен становиться боевой HUD-сценой этой фичи.

#### Runtime Source of Truth

- Все изменяемое боевое состояние должно жить не россыпью по полям `CombatScene`, а в отдельном `CombatRuntime` или `CombatState`.
- `CombatScene` оркестрирует работу, но не является единственным хранилищем боевых данных.

`CombatRuntime` должен содержать как минимум:

- текущее состояние state machine;
- elapsed time от старта текущего боя;
- elapsed time от старта текущей `wave`;
- `base HP`;
- параметры `record`:
  - текущий угол;
  - предыдущий угол;
  - скорость вращения;
  - стартовый угол;
- структуру слотов:
  - `slot index`;
  - назначенная `pawn`;
  - мировая позиция;
  - центр сектора;
  - текущее состояние визуальной активации;
- текущее состояние `note packet`:
  - `color | null`;
  - `count`;
  - визуальные note instances;
- активных врагов;
- активные `sub-wave`;
- еще не активированные `sub-wave`;
- счетчики оставшихся врагов;
- флаги `victory/defeat`;
- ссылки на runtime-представления VFX и временные боевые эффекты.

#### Combat State Machine

Обязательные состояния:

- `preview`
- `running`
- `paused`
- `victory`
- `defeat`

Правила переходов:

- `preview -> running`
  - после короткой стартовой паузы;
- `running -> paused`
  - по нажатию `Pause`;
- `paused -> running`
  - по нажатию `Resume`;
- `running -> victory`
  - когда закрыты все условия завершения `wave`;
- `running -> defeat`
  - когда `base HP <= 0`;
- `paused -> restart`
  - через полный restart сцен;
- `victory/defeat -> restart`
  - через полный restart сцен.

#### Frame Update Ownership

Бой должен уважать fixed pipeline scaffold `GameScene`:

1. input systems
2. simulation systems
3. root entities
4. object pools
5. cleanup hooks

Для этой фичи рекомендуемый расклад ответственности такой:

- Input phase:
  - чтение `Pause`, `Resume`, `Restart`;
  - возможные service/debug hotkeys, если понадобятся.

- Simulation phase:
  - update state machine timers;
  - update `wave` schedule;
  - activate `sub-wave`;
  - spawn enemies from `spawn bag`;
  - advance `record` angle;
  - process slot crossings;
  - resolve pawn activations;
  - update enemy movement and attacks;
  - resolve death/removal flags;
  - check victory/defeat.

- Root entities:
  - визуальное обновление `record`;
  - синхронизация рендера `pawn`, `note packet`, `base`, врагов;
  - короткие non-authoritative visual tweens, если они привязаны к entity.

#### EventBus Responsibilities

`EventBus` должен использоваться как единственный канал связи между `CombatScene` и `HUDScene`.

Минимально ожидаемые группы событий:

- lifecycle:
  - combat scene ready;
  - HUD scene ready;
  - combat restarted;

- state:
  - combat state changed;
  - pause opened;
  - pause closed;
  - combat ended;

- HUD data:
  - wave info updated;
  - enemies left updated;
  - base HP updated;

- combat feedback:
  - slot activated;
  - note packet changed;
  - enemy spawned;
  - enemy died;
  - base took damage.

Документ не фиксирует точные строковые event id, но фиксирует обязанность держать эти сообщения типизированными и отдельными по смыслу.

### Config Strategy

Все runtime-значения этой фичи должны быть вынесены в конфиги.

Нужны как минимум четыре смысловых блока конфигов:

- `CombatLayoutConfig`
  - координаты и размеры layout;
  - позиции `record`, `base`, `needle`, packet anchor, HUD anchors;

- `CombatBalanceConfig`
  - `base HP`;
  - скорость вращения;
  - таблица множителей `finisher`;
  - placeholder-числа урона и статов врагов;

- `CombatContentConfig`
  - `pawn` definitions;
  - enemy definitions;
  - стартовый preset слотов;
  - `outputNoteColor`;

- `CombatWaveConfig`
  - `wave`;
  - `sub-wave`;
  - `spawn bag` content;
  - интервалы и окна спавна.

Координаты layout должны храниться в пикселях, а не в процентах экрана.

### Layout and Spatial Decisions

#### Viewport

- базовый viewport: `720x1280`
- target orientation: `9:16 portrait`

#### Record Geometry

Стартовые placeholder-значения:

- `recordCenterX = 360`
- `recordCenterY = 1140`
- `recordRadius = 420`

Следствия:

- пластинка почти целиком шириной экрана;
- нижняя часть круга скрыта за нижней границей;
- скрыто примерно около трети диаметра;
- верхняя часть пластинки остается видимой и читаемой;
- центр базы оказывается низко, почти у нижней части экрана.

#### Enemy Zone

- Вражеская зона должна занимать примерно половину экрана по высоте.
- Враги не должны заходить на пластинку.
- Между линией визуального давления и верхней кромкой видимой пластинки должен оставаться понятный зазор.

#### Base Block

Стартовые placeholder-значения:

- `baseCenterX = 360`
- `baseCenterY = 1140`
- `DJ booth width ≈ 220 px`
- `DJ booth + capybara height ≈ 180 px`
- `base HP bar width ≈ 180 px`
- `base HP bar height ≈ 16..20 px`
- `note packet anchor ≈ 70..90 px` над центром базы

Эти числа не являются каноном и должны легко тюниться в конфиге.

### Rendering Model

#### What Rotates

Вместе с пластинкой вращаются:

- виниловый диск;
- секторные рамки;
- графика `inner rule zone`;
- rule text;
- pedestal;
- декоративная графика слота.

Статичны поверх пластинки:

- `base / DJ booth / capybara`;
- `base HP bar`;
- `needle`;
- `note packet`;
- верхний HUD;
- overlay;
- будущие кнопки `Rewind / FastForward`.

#### What Stays Upright

Физически с пластинкой вращаются:

- rule text;
- графика правил;
- pedestal.

Всегда upright остаются:

- `pawn construct`;
- `tier`-звезды под `pawn construct`.

Это осознанный гибрид:

- слот должен ощущаться как часть физического винила;
- боевая фигура должна оставаться читаемой.

#### Layering / Depth

Стартовый порядок слоев снизу вверх:

1. background / backdrop
2. enemy lane decorations
3. `record` base disc
4. slot frames + rule labels + pedestals
5. backplates будущих time-control кнопок
6. `base / DJ booth / capybara`
7. `pawn constructs`
8. `note packet`
9. combat VFX
10. top HUD
11. pause/result overlay

Требования:

- слои должны быть именованными;
- глубины не должны быть разбросаны магическими числами;
- порядок должен легко переопределяться позже.

#### Enemy Sorting

- Враги могут накладываться.
- Для них нужен `Y-sort`.
- Кто ниже по экрану, тот рендерится выше.
- Враг должен быть представлен одним render unit:
  - body;
  - HP bar;
  - локальный hit feedback.
- Сортировка должна идти на уровне enemy container, а не отдельных частей.

### Combat Simulation Details

#### Robust Slot Crossing Processing

Нельзя ограничиваться “максимум одна активация за кадр”.

Система обязана:

- хранить `previousRecordAngle` и `currentRecordAngle`;
- вычислять все slot crossings между ними;
- активировать все пересекшиеся слоты в правильном порядке.

Причины:

- защита от lag spikes;
- подготовка к будущему `FastForward`;
- честное непрерывное вращение без потери событий.

#### Activation Sequence

Для `generator`:

1. слот пересекает `needle`;
2. запускается trigger feedback слота;
3. выбирается цель;
4. наносится hit по врагу;
5. в центр летят `2` новые ноты;
6. `note packet` обновляется.

Для `finisher`:

1. слот пересекает `needle`;
2. запускается trigger feedback слота;
3. текущий `note packet` летит из центра к `finisher`;
4. рассчитывается число поглощенных нот;
5. наносится hit по врагу;
6. обратно в центр летит `1` новая выходная `note`;
7. `note packet` обновляется.

Для `empty slot`:

1. срабатывает только pulse сектора;
2. логика урона и нот отсутствует.

#### Color Break Sequence

При конфликте цвета нужен явный feedback:

- старый пакет визуально ломается;
- старые ноты гасятся или рассыпаются;
- создается новый пакет по правилам активированной пешки;
- в будущем сюда будет добавлен звук `scratch` и короткий ducking музыки.

### VFX Technical Approach

Первая версия VFX строится без шейдеров и custom pipelines.

Разрешенные строительные блоки:

- заранее сгенерированные текстуры;
- `Image` / `Sprite` / `Container`;
- `tweens`;
- `particles`;
- `blendMode: ADD`;
- ограниченное использование `Graphics`.

Рекомендуемый набор эффектов:

- slot trigger pulse;
- pawn glow pulse;
- note flight;
- beam hit;
- enemy hit flash;
- base hit flash;
- victory / defeat overlay emphasis.

Рекомендуемый VFX orchestration:

- отдельный `CombatVfxSystem`;
- эффекты вызываются по смыслу, а не размазываются по логике `pawn` и врагов;
- часто используемые объекты совместимы с pooling.

### Primitive Asset Strategy

Визуал должен строиться через runtime-generated textures из примитивов, а не через живую перерисовку большого объема `Graphics` каждый кадр.

Это особенно полезно для:

- note glyph;
- pedestal parts;
- simple enemy bodies;
- slot frame elements;
- hp bar pieces;
- glow sprites;
- beam fragments.

`Graphics` допустимы:

- для debug-вспомогательных линий при локальной разработке;
- для отдельных beam-полилиний;
- для простых transient mask/flash случаев.

### Logging and Diagnostics

- На старте не нужен полноценный debug overlay.
- Для диагностики достаточно:
  - `console.warn`;
  - `console.log`;
  - структурированных сообщений об инвариантах.

Особенно полезно логировать:

- пропущенные или странные crossings;
- неконсистентные состояния `sub-wave`;
- invalid `outputNoteColor`;
- переполнение `note packet`;
- `Restart` без полной очистки runtime, если такое случится.

## UI / UX

### Screen Composition

- Экран проектируется под `9:16 portrait`.
- Enemy zone должна занимать примерно половину экрана.
- Пластинка должна быть показана частично, а не целиком.
- Нижняя часть пластинки уходит за экран.
- База находится в центре пластинки и сидит низко, почти у нижней части видимого экрана.
- Враги никогда не заходят на пластинку.
- Пластинка остается в первую очередь понятным боевым UI-элементом.

### Top HUD

- Слева находится `Pause`.
- По центру находится `Wave 1/1`.
- Рядом допускается компактный `Enemies left`.
- Таймер волны не показывается.
- `Base HP` не выносится в верхний HUD.

### Base UI

- `Base HP` показывается как обычный горизонтальный HP-бар прямо под базой.
- `note packet` висит прямо над головой капибары.
- Это решение считается каноном для первой итерации.

### Future Time Control Placement

- `Rewind` и `FastForward` не входят в эту фичу как механика.
- Но их будущее место уже зарезервировано композиционно:
  - они должны сидеть поверх нижней видимой части пластинки;
  - у них должна быть собственная подложка;
  - под них не нужна отдельная съедающая экран нижняя панель.

## Art Direction

### Overall Style

- Визуальный язык: `graphic neon schematic`.
- Сцена должна ощущаться как стильная неоновая боeвая схема, а не как богатая иллюстрация.
- Арт собирается из примитивов и generated textures.
- Главный приоритет: читаемость, силуэт и настроение.

### Record and Slots

- `Record` — главный композиционный якорь сцены.
- Секторы должны читаться отчетливо.
- Пустой слот тоже должен оставаться видимым.
- Внутри `inner rule zone` используется буквальный schematic language:
  - `generator`: `+♪♪`
  - `finisher`: `-all ♪ -> +♪`
- Надпись сначала пробуем класть по дуге.
- Если это неоправданно сложно в Phaser, допускается прямой fallback.

### Pawn Visual Identity

- `pawn` визуально состоит из:
  - физического pedestal, привязанного к пластинке;
  - holographic 2.5D construct над pedestal.
- У каждой конкретной `pawn` собственный silhouette.
- При этом у семейства `generator` должен читаться язык “эмиссии наружу”.
- У семейства `finisher` должен читаться язык “сбора и разряда”.
- `Tier` показывается золотыми звездами под construct.

### Base / DJ Booth / Capybara

- Центральная база минималистична.
- Нельзя пытаться примитивами “нарисовать богатую сценку”.
- Нужно различать:
  - сам `DJ booth`;
  - милый mascot-силуэт капибары.
- Капибара должна читаться как cute mascot, а не как детализированное животное.

### Enemies

- Все три цветовых врага принадлежат к одному семейству.
- У них общий базовый archetype.
- Различия между цветами читаются:
  - цветом;
  - glow;
  - silhouette accent.

## Audio / SFX

- На первую реализацию звук не обязателен.
- Документ фиксирует будущий hook:
  - при `color break` должен появиться неприятный `scratch`;
  - музыка должна на короткое время приглушаться.
- Пока без звука визуальный feedback обязан уже сам по себе объяснять, что произошел разрыв цветовой цепочки.

## Performance / Technical Constraints

- Базовый target viewport: `720x1280`.
- Layout и боевые числа должны быть вынесены в конфиги.
- Generated textures предпочтительнее постоянной живой геометрии в кадре.
- Часто используемые VFX должны быть совместимы с pooling.
- `CombatScene` не должна превращаться в giant god object, держащий и симуляцию, и UI, и эффектные детали без разделения ответственности.
- Первая версия не использует:
  - шейдеры;
  - custom pipelines;
  - сложный post-processing.

## Edge Cases & Error States

- Crossing нескольких слотов за один кадр не должен теряться.
- Переполнение `note packet` не должно ломать пакет и не должно поднимать `count > 5`.
- Активация `pawn` без доступной цели не должна бросать ошибку и не должна стопорить боевой цикл.
- `finisher` без подходящих нот обязан все равно бить по нулевому множителю, а не пропускать активацию.
- Одновременная работа нескольких `sub-wave` не должна нарушать детерминированность их собственных интервалов.
- Временное отсутствие живых врагов на поле не должно вызывать `victory`, если впереди есть будущие `sub-wave`.
- Вражеский `Y-sort` не должен ломаться при наложении контейнеров.
- Если rule text по дуге окажется слишком дорогим, допускается прямой fallback без изменения всей системы слота.
- Полный restart должен удалять старые таймеры, старые enemy instances, старые VFX и старый runtime state.

## Game Content

### Record

- Роль: главная боевая поверхность.
- Ключевые параметры:
  - `8` слотов;
  - непрерывное вращение;
  - стартовый угол в конфиге.

### Needle

- Роль: фиксированная точка активации.
- Ключевые параметры:
  - позиция `12:00`;
  - вертикальная ориентация вверх;
  - визуальная связь с центром базы.

### Base / DJ Booth / Capybara

- Роль: объект защиты и центральный визуальный якорь.
- Ключевые параметры:
  - `base HP`;
  - горизонтальный HP-бар под базой;
  - anchor для `note packet`.

### Note

- Роль: переносимый боевой ресурс.
- Ключевые параметры:
  - единый glyph;
  - перекраска по цвету;
  - использование и в полете, и в UI-представлении.

### Pawn Definitions

Стартовый набор:

- `red generator`
- `green generator`
- `blue generator`
- `red finisher`
- `green finisher`
- `blue finisher`

Общие параметры:

- `id`
- `type`
- `color`
- `baseDamage`
- visual silhouette key
- pedestal/VFX family key

Дополнительный параметр для `finisher`:

- `outputNoteColor`

Стартовый placeholder mapping:

- `red finisher -> output green`
- `green finisher -> output blue`
- `blue finisher -> output red`

### Enemy Definitions

Стартовый набор:

- `red enemy`
- `green enemy`
- `blue enemy`

Общие стартовые placeholder-числа:

- `maxHp = 30`
- `moveSpeed = 70 px/s`
- `attackRange = 170 px`
- `attackCooldownMs = 1000`
- `attackDamage = 5`

### Base Placeholder Numbers

- `baseMaxHp = 100`

### Pawn Placeholder Numbers

- `generator baseDamage = 10`
- `finisher baseDamage = 20`

### Wave

- В первой версии одна `wave`.
- У нее есть `startingRecordAngle`.
- Она состоит из набора `sub-wave`.

### Sub-wave

- У каждой `sub-wave` есть:
  - `startTimeMs`
  - `spawnIntervalMs`
  - `enemyId -> count`

Для первой итерации допустим короткий baseline порядка `5..7` врагов в сумме, но это остается тюнимым контентом.
